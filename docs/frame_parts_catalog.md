# 仕様書:フレームパーツカタログ(初期セット)

- 版数:v2.0(企画書 Draft v0.5 別紙2)
- 作成日:2026-07-22(v1.0)/ 改訂:2026-07-22(v2.0)
- 位置づけ:別紙1で決定した案A(既製パーツ・フルスナップ方式)のMVP用フレームパーツカタログを具体化する。数値は開発着手時の初期データ(JSON)にそのまま落とせる粒度で定義する。
- **v2.0改訂の要旨:** 接続仕様を「接続点の型(G/S/H/A/W/P)」方式から**「穴とピン」の統一文法**に全面刷新(企画者提案による。参考:[リンク機構シミュレータ](https://edu-mat.sakuraweb.com/linksim/linksim.html?lang=ja)の穴基準・回転可能な組立方式)。これに伴い**受動関節(ピン1本の回転)が解禁され、リンク機構(からくり)が組める**ようになった。§2・§5・§7を全面改訂、三角プレートを追加(24→25種)。

## 1. 設計原則

1. **実在規格に接地する。** 架空の寸法体系を発明せず、ホビーロボット・工作の現実に流通する規格に寄せる。具体的には**穴ピッチ5mmのグリッド**(タミヤ ユニバーサルプレート系の事実上の標準)と、**市販のアルミサーボブラケット**(SG90用・標準サーボ用マルチファンクションブラケット・ハイパワーサーボ用)を範とする。これにより「実機化の夢」(企画書§8将来構想)への接続を保つ。
2. **構造材の電力コストは0。** 「電気を使うものだけが電力コストを持つ」というルールの一貫性を守る。構造材が機体に課すのは質量だけ。
3. **材質は2系統、どちらにもゲーム上の意味を持たせる。**
   - **プラ材**(ABS相当、密度1.2 g/cm³):軽い。標準の選択肢。
   - **アルミ材**(密度2.7 g/cm³):重いが見た目が良い。そして**「重さがバラスト(おもり)として役に立つ」**という積極的な使い道を持つ。低い位置にアルミ材を使って重心を下げる、後ろに使って前のめりを直す——強度計算を持たない本アプリで、重い材料をあえて選ぶ理由を重心メカニクス側に作る。
4. **質量は形状から機械的に導出する。** 質量=体積×密度×穴あき係数0.7(5mmピッチのΦ3穴で約3割抜ける実測近似)。カタログには計算済み公称値を保持し、案B(パラメトリック化)移行時も同じ式で連続的に計算できるようにする。
5. **パーツ数は「少なくて豊か」を狙う。** LEGOの基本ブロック思想に倣い、MVPは約24種に絞る。組合せで多様性が出る直交的なラインナップを優先し、用途が重複するパーツは入れない。

## 2. 接続規格:「穴とピン」の統一文法(v2.0)

接続点の「型」は廃止する。代わりに、**すべてのパーツは5mmピッチの「穴」を持ち、重ねた穴に「ピン」を挿すことだけで接続する**。ルールは2行で説明できる。

> **ピン1本 → そこを中心にくるくる回る(回転関節)**
> **ピン2本以上 → 固定(剛結合)**

物理的に自明(1点留めは回る、2点留めは回らない)なので、子どもに説明が要らない。工作の割りピンやリンク機構シミュレータと同じ、手で知っているルールである。

### 2.1 穴の仕様

- 全パーツ(フレーム、プレート、サーボホーン、サーボ胴体の取付フランジ、パワーボックス底面、車輪ハブ、おもり、かざりの足)は5mmピッチグリッド上の穴を持つ。**穴は1種類だけ**で、どの穴とどの穴でも重なれば接続できる。
- 実在のサーボも実際にはフランジの取付穴で固定し、ホーンの周囲穴で腕を留める。つまり「ぜんぶ穴」は抽象化であると同時に現実そのままでもある。
- ピンはパーツではなく**操作**(重なった穴をタップ/クリックで留める)。質量は0として扱う(実機ならネジ+ナット数gだが、ゲームの見通しを優先して無視。明記しておく)。

### 2.2 特別な穴は2つだけ(内部実装上の区別。UI上は同じ「穴」に見える)

| 特別穴 | ある場所 | ピンを挿すと |
|---|---|---|
| **駆動穴** | サーボホーンの中心/車輪用サーボの軸 | サーボが回す**能動関節**になる(操作割当の対象) |
| アイドラー穴 | 標準・ハイパワーサーボの背面軸 | 対応するホーン中心と同一関節の従動支持(両持ち) |

それ以外の穴同士の接続はすべて「ピン1本=受動関節/2本以上=剛結合」の一般則に従う。

### 2.3 受動関節の解禁=リンク機構(からくり)遊び

ピン1本の受動関節が使えることで、**サーボ1個+骨組みだけでリンク機構が組める**。

- 4節リンクの脚(サーボ1個でカシャカシャ歩く多脚)、パンタグラフ、1サーボで開閉するグリッパー、クランクで羽ばたく飾り——「からくりの知恵」で動きを作れる。
- **電力コストのゲームに第3の選択肢が生まれる:** 強いサーボを増やす(ボックス肥大)か、軽さで逃げるか、**リンク機構で少ないサーボから動きを取り出すか**。工夫がそのまま軽量化になる、本企画の思想と完全に整合する拡張。
- 初心者が受動関節だらけの「ぐにゃぐにゃロボ」で迷子にならないよう、組立支援を置く(§7.2)。

### 2.4 スナップ実装の要点

- パーツを近づけると、重なりうる穴のペアがハイライトされ、グリッドに吸着する(面同士が平行に重なる配置のみ接続可。3D空間での穴軸合わせはスナップが担い、ユーザーに軸を意識させない)。
- 留めた箇所は「回る(ピン1)/固定(ピン2+)」がアイコンで常時表示され、タップで切替(ピンの増減)できる。
- 同一機体内のフレーム同士の衝突判定は無効とする(実機なら座金・スペーサで逃がす板厚干渉の問題を、配線省略と同じ思想でゲーム的に割り切る。シミュレータ側も機体内自己衝突はリンク隣接無効化で扱う)。

### 3.3 サーボまわり(3種)— v2.0で大幅簡素化

穴ピン文法の採用で、サーボまわりは劇的にシンプルになった。

- **ホーンはサーボに標準装備**(パーツとして数えない):中心の駆動穴+周囲8穴の円形ディスクとしてモデル化。周囲穴に2ピンで腕を固定すればサーボと一緒に回る。
- **サーボ胴体はフランジ穴で直接取付**:実物と同じく、フランジの穴を骨組みの穴に重ねて2ピンで固定(ゲーム内ではフランジ穴を5mmグリッド整合とみなす)。専用マウント型は不要になった。

## 4. 質量の根拠(実部品との突合メモ)

- プレート類:タミヤ ユニバーサルプレート(160×60×3 mm、ABS、5mmピッチ)の実物系列と密度計算(×穴あき係数0.7)がおおむね一致することを確認済み。
- 標準サーボ用アルミUブラケット:市販のMG996R用マルチファンクションブラケット(1個十数g)に整合。
- ハイパワーサーボ用:STS3215向け市販ブラケット類(20〜30g)に整合。
- 開発時、初期データ投入の際に代表実部品を各1点実測(または販売ページ記載値で再確認)して公称値を確定すること。

## 5. データスキーマ(開発担当者向け)

パーツ定義はすべてJSONデータ駆動(企画書§7)。フレームパーツの定義例:

```json
{
  "id": "FR-B060",
  "category": "frame/beam",
  "displayName": { "kids": "ほね(中)", "adult": "ビーム 60mm" },
  "refRealPart": "5mmピッチ汎用ビーム(タミヤ系互換)",
  "materialOptions": ["plastic", "aluminum"],
  "massG": { "plastic": 2, "aluminum": 4 },
  "powerCost": 0,
  "geometry": { "type": "box", "sizeMm": [60, 10, 3] },
  "collision": { "type": "box", "sizeMm": [60, 10, 3] },
  "holes": { "pattern": "grid", "pitchMm": 5, "rows": 1, "cols": 12, "originMm": [-27.5, 0, 0], "normal": [0, 0, 1] }
}
```

サーボ側は`holes`に加えて特別穴を持つ:

```json
"specialHoles": [
  { "kind": "drive", "posMm": [0, 9.5, 15], "axis": [0, 1, 0] },
  { "kind": "idler", "posMm": [0, -20, 15], "axis": [0, 1, 0] }
]
```

機体モデル側の接続レコードは「どの穴とどの穴を、何本のピンで留めたか」だけ:

```json
{ "partA": "p12", "holeA": [0, 3], "partB": "p07", "holeB": [1, 0], "pins": 1 }
```

実装上の要点:

- `holes`の座標系定義がスナップ実装とURDF/MJCFのフレーム変換の共通ソースになる(二重管理しない)。接続点の「型」テーブルは存在しない。
- 書き出し時の変換則:**pins≧2の接続で結ばれたパーツ群を1リンクに合成**(質量・重心・慣性を合算。企画書§5.2)。**pins=1は受動回転関節**、**駆動穴への接続は能動関節**としてリンク境界になる。
- 材質切替は`massG`の参照切替のみで、形状・接続は不変(UIも物理も差分最小)。

## 6. 案B(骨格自動生成)への伸びしろ

- 「ほね」系4種は、案B移行時に**長さ連続可変のパラメトリックビーム**へ昇格する(30〜120mmスライダー)。ID体系は`FR-B{length}`として連続値を許すよう設計しておく。
- 「いた」系は、案Bの**パネル自動提案(骨組み区画への肉付け)**の雛形になる。
- つまり本カタログは使い捨てではなく、案Bのパラメトリック雛形の離散サンプル版という位置づけ。データスキーマの`geometry`をパラメトリック定義で持っておくのはそのため。

## 7. リンク機構(からくり)の扱い(v2.0新設)

### 7.1 エクスポートと閉ループ問題

4節リンクなどのリンク機構は運動学的な**閉ループ**を含む。ここに形式間の重要な差がある。

- **MJCF:** `equality/connect`制約で閉ループを表現可能。**シミュレータ主経路(MJCF)では、からくりは問題なく動く。**
- **URDF:** 木構造しか表現できず、閉ループを持てない(形式仕様上の制限)。リンク機構を含むモデルのURDF出力は**ループの1箇所を切って出力し、「外部ツールでは機構が正しく動かない場合がある」旨をエクスポート時に表示**する。
- P0検証項目に「MuJoCo(WASM)で4節リンクが等式制約で動くこと」を追加すべき(企画書§8に反映)。

### 7.2 組立支援(「ぐにゃぐにゃロボ」対策)

受動関節の解禁は自由度が増えるぶん、初心者が意図せずぶらぶらの機体を作るリスクがある。3つの支援を置く。

1. **デフォルトは固定。** 穴を重ねて留めると、まず2ピン固定で吸着する。「回る」はワンタップで切り替える明示的な選択にする(回したい人だけが回す)。
2. **ぶらぶらメーター。** 機体の受動自由度を数えて表示(「ぶらぶらの関節が3つあるよ」)。リンク機構として閉じている受動関節はカウントから除外されるので、正しく組めたからくりは「ぶらぶら0」になる——機構が成立した瞬間が視覚的にわかる。
3. **からくりレシピ。** 4節リンクの脚、1サーボグリッパー、パンタグラフ等の組み方をレシピカード(テンプレート)として提供。まず動くものを置いてから改造する導線。

## 8. 残論点

1. **グリッドピッチの最終確認。** 本仕様は5mm(タミヤ互換)で確定を提案。3Dプリント時代の別規格(例:M3系メカ規格)もあるが、日本の教育現場での入手性・馴染みでタミヤ系が優位と判断。
2. **初期セットの過不足。** 25種は「4脚・2輪車・アーム・2脚がすべて組める最小集合+からくり素材」として設計したが、プレイテストで「作りたいのに作れない形」が出たら追加する(データ駆動なので追加コストは小さい)。
3. **「あしうら」の摩擦パラメータ。** 歩行体験の質に直結するため、P0のMuJoCo検証項目に含めたい(シミュレータ側と要調整)。
4. **リンク機構(受動関節)をMVPに含めるか(新規)。** 穴ピン文法自体はMVPから入れる(実装の土台は同じ)。ただし受動関節の解禁をMVPからにするか、まず全接続を固定限定で出してP2で解禁するか——閉ループ検証(§7.1)の結果とチュートリアル設計を見て判断したい。
5. **ピン質量0の妥当性。** ゲームの見通し優先で0とした。実機化リストにはネジ・ナットとして計上する(将来)。
